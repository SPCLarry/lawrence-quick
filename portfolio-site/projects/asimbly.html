Asimbly: Scalable FPS Systems

Designing a robust architecture for Character Movement, UI Persistence, and Interactions in a long-term collaborative project.

At a Glance

  • Role: Gameplay Engineer
  • Team: 2 Developers
  • Duration: 1 Year+

Contents

1. Character Controller 2. UI Architecture 3. Materials & FX 4. Interaction System

1. Character Systems

The core requirement was a "True First Person" controller that supported complex states like sliding and mantling without becoming a tangle of boolean checks.

The Solution: State Machines

I extended UCharacterMovementComponent to handle custom movement modes. Transitions are handled via a strict Enum State Machine. This prevents invalid states (e.g., trying to slide while in mid-air).

For mantling, I utilized a "Capsule Trace" followed by a "Sphere Trace" to detect ledge height and depth, ensuring the player only mantles onto valid geometry.

2. UI Architecture

A common issue in Blueprint projects is UI data loss when levels change. I implemented a GameInstanceSubsystem in C++ to act as the persistent data layer.

The UI Widgets act purely as a "View." When opened, they query the Subsystem for the current Sensitivity or Volume settings. This creates a clean separation of concerns and makes debugging much easier.

3. Materials & Visuals

I created a custom world grid material extended from third-party assets and HLSL-driven UI visual effects.

4. Interaction System

To avoid hard references (casting to specific actors), I utilized a Blueprint Interface BPI_Interact.

Event Interact(PlayerRef)
Any actor that implements this interface responds to the player's input. This allows for a decoupled system where doors, switches, and pickups all share logic without the player character needing to know what they are.
