Asimbly

Role: Gameplay & Systems Developer | Engine: Unreal Engine 5


Overview: Asimbly is a long-term, two-person collaborative project. My role focused on building scalable, maintainable gameplay systems including character movement, UI architecture, and AI behavior.

System Breakdowns:

  • 1. Character Systems
  • 2. UI Systems
  • 3. Material Design & HLSL
  • 4. Sound Systems
  • 5. AI Design
  • 6. Interaction Systems

1. Character Systems

Scope

Implemented a True First Person character setup extending Unrealâ€™s CharacterMovementComponent. Features include sliding, mantling, and climbing.


Technical Focus

The priority was responsiveness and clean state transitions. I utilized a custom Enum state machine to handle transitions between Walking, Sliding, and Mantling to prevent state conflicts.

Reflection: Early iterations relied too heavily on Blueprint tick for state checking. I refactored this to event-driven logic to optimize performance and improve frame-rate stability.

2. UI Systems

Scope

Designed a persistent Settings system (Keybindings, Audio, Graphics) and a modular Main Menu.


Architecture

I created a custom C++ GameInstanceSubsystem to handle saving/loading user configuration. This separates UI logic (View) from data storage (Model), making the system accessible from any level without casting overhead.

Reflection: Separating the Data Layer from the Widget Layer was crucial. Initially, widgets held their own variables, which caused data loss on level transition. The Subsystem solved this.

3. Material Design & HLSL

Scope

Created a custom world grid material extended from third-party assets and HLSL-driven UI visual effects for damage feedback.


Technical Focus

Focused on visual clarity and shader performance. UI effects use dynamic material instances to react to gameplay events in real-time.

4. Sound Systems

Environmental audio feedback driven by physics materials and a persistent music manager.


The system queries the Physical Material of the surface the player walks on to trigger the correct footprint SFX. The music system persists across level loading boundaries using the GameInstance.

5. AI Design

Scope

Implemented Behavior Trees and EQS (Environment Query System) for tactical AI decision making.


Focus Areas

Clean behavior separation. The AI uses a modular Service to check for line-of-sight and updates the Blackboard only when necessary, reducing overhead.

6. Interaction Systems

Scope

A generalized interaction interface and a radar system for locating objectives.


Architecture

Used a Blueprint Interface (BPI_Interact) so the player character does not need to cast to specific actors. Any actor implementing the interface responds to the "Interact" message.

Reflection: This was a shift from my early projects where I used casting. The Interface approach decoupled the systems, allowing me to add new interactable types without touching the player code.
